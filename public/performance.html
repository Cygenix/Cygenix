Batch and chunk sizes
Controls how many rows are fetched or inserted in each operation
Fetch page size
Number of rows fetched from the source database per page. Larger values mean fewer round-trips but more memory. Reduce if you hit memory errors or timeouts on wide tables.
Guidance: 2,000 is a safe default. Use 500-1,000 for tables with many large text/blob columns. Use 5,000+ for narrow tables on fast local networks.
rows per page
Insert batch size
Rows per INSERT statement. SQL Server supports up to 1,000 per INSERT VALUES. Smaller batches are safer; larger batches are faster but harder to roll back on failure.
Guidance: 500 is recommended. Reduce to 100-200 for wide rows (>50 columns) or large NVARCHAR fields that approach the 8,060 byte SQL Server row limit.
rows per INSERT
Concurrent batch threads
How many INSERT batches are sent in parallel within each page. Parallelism speeds up inserts significantly but increases lock contention. Set to 1 for sequential (safest).
Caution: Values above 4 can overwhelm Azure SQL Basic/Standard tiers. Always test with 2 first and monitor DTU usage before increasing further.
parallel threads

Timeouts
How long to wait before abandoning a connection or query
Connection timeout
Maximum wait when establishing a new database connection. Increase for slow VPNs or Azure SQL cold-starts. Reduce to get faster error feedback on wrong credentials.
Guidance: 30s is standard. Azure SQL serverless may need 60-90s on cold start. On-premises over LAN rarely needs more than 15s.
seconds
Query / execution timeout
Maximum time a single SQL statement may run before cancellation. Long migrations or complex validations need higher values. Set to 0 to disable (not recommended in production).
Guidance: 300s (5 min) is safe for most migrations. For 1M+ row tables consider 600-1200s. Never set 0 in shared environments.
seconds (0 = none)
Azure Function / API timeout
Maximum time the browser waits for the Azure or Netlify Function connector. This must be higher than your query timeout or the browser will cancel a query still running on the server.
Important: The Azure Function host itself has a separate timeout (default 30s on Consumption plan, 5 min on Premium). If hitting 30s limits, upgrade to Premium or set functionTimeout in host.json.
seconds
Retry attempts on failure
How many times to retry a failed batch before marking it as an error. Useful for transient network blips or Azure throttling (HTTP 429). Each retry waits 2x longer than the last (exponential backoff).
retries

Data loading behaviour
Controls how data is read, written and validated during migration
Use NOLOCK (READ UNCOMMITTED) on source reads
When reading source data, use READ UNCOMMITTED to avoid blocking writers. This can cause dirty reads (reading uncommitted rows) but reduces lock contention on live source databases significantly.
Guidance: Enable only if source DB is live under active write load. Disable for offline/static sources to guarantee consistency.
OFF
Wrap inserts in transactions
Each batch of inserts is wrapped in BEGIN TRAN / COMMIT TRAN. A failed batch only rolls back that batch, not the whole migration. Disable only for maximum speed on targets without good transaction support.
ON
SET NOCOUNT ON
Suppresses row-affected messages from SQL Server. Reduces network traffic for large batch operations and can measurably improve throughput on high-latency connections.
ON
Suggest disabling non-clustered indexes
Adds ALTER INDEX DISABLE / REBUILD comments around the generated migration SQL. Disabling non-clustered indexes before a large insert then rebuilding afterwards can be 3-10x faster. These are added as comments only — you run them manually in SSMS before and after the migration.
Note: Never disable the clustered index — this makes the table inaccessible. Non-clustered only. Always rebuild before going live.
OFF
Row count verification after each batch
After each page of inserts, runs a COUNT(*) on the target to verify the expected rows were written. Small overhead per batch but confirms no data was silently dropped.
OFF

Schema and connection
How schemas are loaded and connections are managed
Schema cache lifetime
How long table names, columns and types are cached in the browser before re-fetching. Longer cache means faster page loads. If your schema changes frequently during development, use a shorter value.
Max tables in schema drop-downs
Databases with hundreds of tables can slow the browser when populating selectors. This cap limits the initial load — you can still search for any table by typing. Set to 0 for no limit.
tables
Connection pool size
Maximum simultaneous database connections the Azure Function maintains per request. Higher values allow more parallel queries but consume more server resources and DTUs. Most migrations need no more than 4.
connections

Browser and UI
Settings that affect how Cygenix behaves in the browser
Max rows in SQL Editor results grid
Caps rows rendered in the results grid after a SELECT. All rows still return from the server — this only limits display to prevent the browser becoming unresponsive with very large result sets.
rows in grid
Auto-save interval
How often the mapper and editor auto-save your work to localStorage. Lower values mean less risk of losing work but more frequent writes. Set to 0 to disable auto-save.
Progress update frequency
How often the progress bar and row counters update during execution. More frequent updates are easier to monitor but add slight overhead. For very fast connections, increase the interval to avoid UI flickering.
