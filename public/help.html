Cygenix Help Guide

Cygenix is an AI-powered database migration platform. It connects to your source and target databases, maps columns intelligently using Claude AI, applies value translations (Was/Is), and executes migrations with full rollback capability.

🔗 Configure

Connect source & target DBs, map columns with AI, preview SQL and save migration jobs.

📋 Conversion Project

Sequence multiple migration jobs and SQL scripts into a single executable project.

✅ Validation

Run pre/post migration validation scripts to verify data quality and integrity.

Getting Started

The typical migration workflow in Cygenix follows this sequence:

Run validation scripts (pre-migration)

Use the Validation page to identify data quality issues in the source database before migrating.

Configure your migration

Connect to source and target databases, select tables, map columns using Claude AI, and save the job to your library.

Build a Conversion Project

Combine multiple migration jobs and custom SQL scripts into an ordered project and run them as a single job.

Review the Conversion Report

Export an Excel report showing migration results, column mapping, Was/Is rules applied, and any warnings.

Run validation scripts (post-migration)

Re-run validation against the target database to confirm data migrated correctly.

Configure (Live DB Migration)

The Configure page is a 5-step wizard for setting up a migration between a source and target database. Access it from Configure in the sidebar.

Step 1 — Source Database

Connect to the database you are migrating from. Two connection modes are available:

  • Direct connection string — uses the Netlify serverless function to connect. Use standard SQL Server format: Server=host;Database=db;User Id=user;Password=pass;Encrypt=true;
  • Azure Function — connects via the Cygenix Azure Function using Managed Identity. No passwords needed. Enter the Function URL and key.

Click 🔌 Connect source DB to test the connection and load the database schema.

Step 2 — Target Database

Connect to the database you are migrating into. Same two connection modes as Step 1. The Azure Function mode is recommended for Azure SQL as it uses Managed Identity authentication.

Tip: Your connection details are remembered for the session. On future visits the fields will pre-fill from your last session.

Step 3 — Select & Map

Choose the source table and target table from the dropdowns. Claude AI will automatically map columns between the two tables based on name matching and data type compatibility.

  • auto — exact name and type match
  • review — name match but type differs, or similar name
  • unmatched — no suitable target column found

You can override any mapping using the dropdown in each row. Use Re-map with Claude to ask the AI to remap after you've made changes.

If you have a Was/Is translation file, load it here using the Was/Is panel. Rules are shown as 🔄 badges on affected columns.

Step 4 — Preview SQL

For source tables from a live database, click Fetch rows from source DB to retrieve the data. Cygenix then generates batched INSERT INTO statements with all column mappings and Was/Is translations applied.

Three tabs are available:

  • INSERT statements — the full migration SQL
  • Schema check SQL — queries to verify the target schema
  • Verify queries — row count comparison queries

You can download the SQL to run manually in SSMS if preferred.

Step 5 — Save & Finish

Enter a job name and choose what to do next:

  • 💾 Save & go to Dashboard — saves the job and returns to the dashboard
  • 💾 Save & map another table — saves and returns to Step 3 keeping your connections active
  • 💾 Save & open Conversion Project — saves and goes to the project builder
Tip: The job is saved to your browser's local storage and appears in All Jobs and the Conversion Project job library. The full column mapping and Was/Is rules are saved with the job.

Conversion Project

The Conversion Project Builder lets you sequence multiple migration jobs and custom SQL scripts into a single executable project.

Adding Steps

The left panel shows all saved migration jobs. Click any job to add it as a step. Steps show their step number, icon, source → target route, and current status.

Use + Add custom SQL step at the bottom of the left panel to add a SQL script step. Each SQL step can be set to run on either the source or target database — useful for pre-migration cleanup, creating indexes, or archiving processed rows.

Drag steps to reorder them. Click ✕ to remove a step.

Running a Project

Set your source and target connections at the top, click 🔌 Connect source and 🔌 Connect target to verify, then click ▶ Run project.

Steps run sequentially. Status indicators show progress:

  • ⚪ Pending — not yet run
  • 🟡 Running — currently executing
  • 🟢 Passed — completed successfully
  • 🔴 Failed — error occurred
  • 🟠 Skipped — user chose to skip after failure

If a step fails, execution stops and you are presented with three options: ↩ Retry, ⏭ Skip, or ⏹ Abort. Use ⏹ Stop in the topbar to halt execution at any time.

After completion, click 📄 View Conversion Report to open the full Excel-exportable report.

Data Validation

The Validation page reads scripts from a validation_scripts table in your source database and executes them to identify data quality issues.

ColumnDescription
nameScript identifier name
captionHuman-readable description of what the script checks
sqlThe SQL query to execute. Any rows returned = validation failed.
OrderExecution order
codeScript code reference
Script_DestinationDatabase1, 2, or 3 = run on Source. 4 = run on Target.
groupLogical grouping of scripts (e.g. AP-Transactions, GL-Setup)

Pass/Fail rule: If a script returns 0 rows — it passes. If it returns any rows — it fails and those rows are shown in the Excel export so you can investigate the exact records causing the issue.

Use the column header filters to narrow down scripts by name, caption, group, destination database, or status. Active filters highlight in amber. Click ✕ clear to reset all filters.

Click the SQL preview in the SQL column to expand the full query inline. Use ▶ Run on individual rows to test specific scripts, or ▶ Run all scripts / ▶ Run selected for bulk execution.

The 📊 Export to Excel button produces a workbook with a Summary sheet, one sheet per failed script showing the actual problem rows, and a Script Errors sheet for any scripts that encountered SQL errors.

Inventory

The Inventory section stores documents, schema files, data dictionaries and notes relevant to the migration project. Files are stored in your browser's local storage.

Supported file types:

  • CSV / TSV — used as source data for file-based migrations
  • Excel (.xlsx) — source data or Was/Is translation tables
  • SQL (.sql) — schema scripts, migration scripts
  • PDF / Word — reference documents, specifications
  • JSON — structured data exports

If a client has uploaded files via the Client Portal, they appear at the top of Inventory in an amber 👤 Client submissions section. Use + Inventory to pull a client file into the main inventory for use in migrations.

Was/Is files: Upload a CSV or Excel file with "Was" and "Is" (old/new) value columns to the inventory, then load it in Step 3 of Configure. Cygenix will automatically translate matching values during migration.

Conversion Reports

After completing a migration (via Configure Step 5 or Conversion Project), click Report from All Jobs or 📄 View Conversion Report from the project runner.

The report page shows a summary of the migration and allows export to Excel. The Excel workbook contains:

  • Summary — project name, user, date, duration, migration statistics, status
  • Migration Results — source table, target table, rows, errors, status per table
  • Column Mapping — every column mapped with source/target table, type, transform, and Was/Is rules applied
  • Was-Is Rules — complete list of value translations applied
  • Execution Log — (project reports) step-by-step log with status and row counts
  • Warnings — any issues flagged during migration

Was/Is Mapping

Was/Is mapping translates specific values during migration — for example converting status codes A → Active, or renaming department codes to match the target system's naming conventions.

Prepare a CSV or Excel file with columns for: field name, old value (Was), new value (Is). Load it in Configure Step 3 using the Was/Is panel on the right side of the column mapping section.

Columns that have Was/Is rules applied show a 🔄 N rules badge. Hover to see a preview of the translations. The rules are applied at execution time and included in the Conversion Report.

Note: Was/Is rules are case-sensitive by default. Ensure the "Was" values exactly match the source data values.

Client Portal

Clients can be given their own Cygenix login that takes them to a simplified portal (/client.html) where they can upload source data files, documents, and notes for the migration analyst.

Setting up a client account

Invite the client

Go to Netlify dashboard → Identity → Invite users → enter the client's email address.

Set the client role

After they accept the invite, find their user in Netlify Identity → Edit → set app_metadata to {"role": "client"}.

Client logs in

When the client logs in at the Cygenix URL they are automatically routed to the client portal, not the analyst dashboard.

Client portal features

  • Upload files categorised as: Source data, Inventory, Document, Schema/SQL, Other
  • Add notes and messages for the analyst
  • View all previous submissions with timestamps

Client uploads appear in the analyst's Inventory tab under a 👤 Client submissions section. The analyst can dismiss items once reviewed or click + Inventory to add a file to the main inventory.

Connection Strings

Cygenix supports two connection methods for both source and target databases:

Direct SQL Server connection string

Standard ADO.NET format:

Server=your-server.database.windows.net;Database=YourDatabase;User Id=username;Password=yourpassword;Encrypt=true;TrustServerCertificate=false;

For on-premises SQL Server with Windows auth:

Server=192.168.1.100,1433;Database=YourDatabase;User Id=sa;Password=yourpassword;Encrypt=true;TrustServerCertificate=true;

Azure Function (Managed Identity)

For Azure SQL databases, use the Cygenix Azure Function which authenticates via Managed Identity — no passwords stored or transmitted:

https://cygenix-db-api-e4fng7a4edhydzc4.uksouth-01.azurewebsites.net/api/db?code=YOUR_FUNCTION_KEY
Tip: The Azure Function URL and key can be found in the Azure Portal → Function App → Functions → db → Get function URL.

Troubleshooting

IssueSolution
Connection failed: Failed to fetch Check the URL is correct. For Azure Function, ensure the ?code= key is included. Check CORS is enabled for the Cygenix domain in the Azure Function App.
Non-JSON response (HTTP 500) The Azure SQL database may be paused (auto-pause on free tier). Wait 20–30 seconds and try again. The first connection after a pause takes time to resume.
Login failed / token error The Managed Identity may not have access to the database. Run: CREATE USER [identity-name] FROM EXTERNAL PROVIDER and grant db_datareader and db_datawriter roles.
Source table shows — in report Save & Finish from Step 5 of Configure (not Execute). The new Save flow correctly records the source table name.
Column mapping not saving Ensure you complete all steps through to Step 5 and click a Save button. Mappings are stored in the job record in browser local storage.
Validation scripts show Non-JSON (500) The Azure SQL database is paused. Click Connect again after 30 seconds to wake it up, then retry loading scripts.
Jobs disappear after refresh Jobs are stored in browser local storage. Clearing browser data will remove them. Export reports before clearing browser storage.
Excel export does nothing SheetJS is loaded on demand from a CDN. Check your internet connection. If behind a corporate firewall, the CDN domain cdnjs.cloudflare.com may need to be whitelisted.
